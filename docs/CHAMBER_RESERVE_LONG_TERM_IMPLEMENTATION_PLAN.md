# ChamberReserve 長期実装計画

作成日: 2026-06-20

## 目的

この文書は、ChamberReserve を完全ゴールまで段階的に実装するための長期計画を定義する。

詳細な画面仕様、API仕様、DB仕様、テストケースは各フェーズの開始文書で個別に定義する。この文書では、全体の到達点、実装順序、フェーズ間の境界、検証方針を固定する。

## 完全ゴール

ChamberReserve の完全ゴールは、社内LAN上で複数の恒温槽を横断的に予約でき、利用者が最小操作で試験条件に合う空き枠を見つけ、管理者が現場運用を止めずに一時利用停止や調整を行える状態である。

完成時には、次が成立していることを目指す。

- 利用者はチャンバーを一台ずつ確認せず、試験条件と必要ブロックから予約候補を探せる。
- 温度サイクル型、固定条件型、自由温湿度型のチャンバー制約を自然に扱える。
- チャンバー内部の 4 x 3 ブロックを視覚的に選択できる。
- 断片化したブロック割り当ては候補に出さない。
- 予約開始、終了、搬入、搬出の制約がチャンバー種別ごとに守られる。
- 予約作成、変更、削除はアカウントなしで行える。
- 4桁PINにより、予約者が自分の予約を変更、削除できる。
- 利用後の予約は削除できず、履歴として残る。
- 管理者は一時利用停止を最優先で登録でき、影響を受ける予約を確認できる。
- TeamPlanner と同じPCで管理しやすいが、URL、ポート、DB、ログ、運用単位は分離されている。
- ローカル運用、バックアップ、復旧、検証の手順が文書化されている。

## 全体方針

### 実装の進め方

実装はフェーズ単位で進める。各フェーズでは、開始前に専用の開始Markdownと貼り付け用 `/goal` を用意する。

各フェーズは、次を満たしてから完了とする。

- そのフェーズの開始文書に書いた完了条件を満たす。
- 必要なテストが通る。
- UIに影響する変更はブラウザで確認する。
- 結果を `PASS` / `FAIL` / `BLOCKED` / `NOT_RUN` で報告する。
- 次フェーズに渡す未決事項を明確にする。

### 実装上の優先順位

優先順位は次の順番とする。

1. 予約可否の正しさ。
2. 利用者が迷わない検索、予約UX。
3. 管理者が影響を把握できる運用性。
4. 社内LANで安定稼働する運用性。
5. レポートや集計などの便利機能。

見た目の美しさは重要だが、予約可否の正しさより優先しない。

### 最初から守る境界

次の境界は、全フェーズを通して維持する。

- 最初の操作はチャンバー選択ではなく条件検索。
- アカウント管理は作らない。
- 仮押さえは作らない。
- 断片化したブロック候補は表示しない。
- メール、Teams、Slackなど外部通知は作らない。
- 奥行き、高さ、重量、通風クリアランス管理は当面作らない。
- TeamPlanner の業務概念は持ち込まない。

## フェーズ構成

## Phase 0: 実装準備と方針固定

### 目的

実装前に、仕様、作業ルール、フェーズ開始文書、Git準備を整える。

### 主な成果物

- 完成仕様書。
- Agent向け実装ガイド。
- `/goal` 実装方針。
- Phase 1 開始文書。
- GitHubリポジトリ設定。

### 完了状態

Codex が Phase 1 実装を開始できるだけの文書とリポジトリ状態が揃っている。

## Phase 1: 単一温度サイクル型チャンバーのUI/UX基盤

### 目的

温度サイクル型チャンバー1台について、利用者が空き検索から予約確定、PIN編集、削除まで行えるUI/UXを先に固める。

このフェーズでは、サーバーAPIやSQLite永続化を作らない。ブラウザ上のモックデータまたはローカル状態で、操作順序、画面構成、入力の少なさ、ブロック選択、候補表示、管理者一時利用停止の見え方を確認する。

### 主な実装範囲

- 最小アプリ基盤。
- UI確認用の単一チャンバーデータ。
- 温度サイクル標準条件。
- 搬入可能 25°C 定常期間、サイクル運転期間、搬出可能 25°C 定常期間。
- 4 x 3 ブロック選択。
- サイズだけ指定、場所まで指定。
- 候補一覧。
- 予約確定。
- 4桁PIN発行。
- PINによる予約変更、削除。
- 利用後削除不可。
- 予約ボード。
- 管理者一時利用停止。
- 一時利用停止による影響予約表示。
- ブラウザ確認用の代表シナリオ。

### 完了状態

ブラウザ上で、単一温度サイクル型チャンバーのユーザー側予約体験と管理者一時利用停止体験が確認できる。

### 次に残すこと

- サーバーAPI、SQLite永続化、最終競合再確認。
- 複数チャンバーの候補検索。
- 固定条件型、自由温湿度型の実装。
- 社内LAN運用の本格化。

## Phase 2: 単一チャンバーのサーバーAPIと永続化

### 目的

Phase 1 で固めたUI/UXを、単一温度サイクル型チャンバーの実データ保存とサーバー側判定に接続する。

### 主な実装範囲

- Node.js TypeScript API。
- SQLite永続化。
- 単一チャンバーの初期データ。
- 予約作成、変更、削除API。
- 4桁PINの保存と照合。
- 予約確定直前のサーバー側競合再確認。
- 利用後削除不可のサーバー側判定。
- 管理者一時利用停止の保存。
- 一時利用停止による影響予約表示。
- UIからAPIへの接続。

### 完了状態

Phase 1 の主要UIフローが、ページ再読み込み後も維持される実データとして動作する。

### 次に残すこと

- 複数チャンバーの候補検索。
- 固定条件型、自由温湿度型のチャンバー種別拡張。
- 社内LAN運用の本格化。

## Phase 3: 複数チャンバー候補検索

### 目的

複数チャンバーを登録し、利用者が条件を入力すると、条件に合うチャンバー候補が横断的に表示される状態にする。

### 主な実装範囲

- 複数チャンバー管理。
- チャンバーごとのブロック構成。
- チャンバーごとの稼働状態。
- 条件に合うチャンバー候補の一覧表示。
- 日程を前後にずらした候補探索。
- チャンバー単位ではなく条件単位の検索UX。

### 完了状態

利用者が複数チャンバーを一台ずつ開かずに、条件に合う空き候補を比較できる。

### 次に残すこと

- チャンバー種別ごとの制約拡張。
- 候補表示のわかりやすさ改善。

## Phase 4: 固定条件型チャンバー対応

### 目的

固定温湿度条件で運用されるチャンバーを予約候補に含められるようにする。

### 主な実装範囲

- 固定条件型チャンバーの登録。
- 固定温度、固定湿度の表示。
- 利用者が変更できない条件のUI。
- 条件一致時だけ候補に出す判定。
- 固定条件型の予約ボード表示。

### 完了状態

固定条件に合う試験だけが固定条件型チャンバー候補として表示される。

### 次に残すこと

- 自由温湿度型の同時利用判定。

## Phase 5: 自由温湿度型チャンバー対応

### 目的

利用者が温度、湿度を指定できるチャンバーを予約候補に含められるようにする。

### 主な実装範囲

- 自由温湿度型チャンバーの登録。
- 温度、湿度入力。
- 同時利用時の完全一致判定。
- 温湿度が異なる予約との競合判定。
- 自由温湿度型の予約ボード表示。

### 完了状態

自由温湿度型では、温湿度条件が完全一致する予約だけが同時利用候補として扱われる。

### 次に残すこと

- 運用管理、履歴、復旧性の強化。

## Phase 6: 社内LAN運用とランタイム管理

### 目的

ChamberReserve を社内LAN上で安定して起動、停止、再起動、状態確認できるようにする。

### 主な実装範囲

- TeamPlanner と分離されたポート、URL、DB、ログ、PIDファイル。
- ローカル起動、停止、状態確認。
- live用DB配置。
- health check。
- 異常終了後の初動手順。
- Windows環境での運用手順。

### 完了状態

TeamPlanner と同じPC上で、混同せずに ChamberReserve を起動、停止、確認できる。

### 次に残すこと

- バックアップ、復旧、履歴管理。

## Phase 7: データ保護、バックアップ、復旧

### 目的

予約データを失わないためのバックアップと復旧手順を整える。

### 主な実装範囲

- 手動バックアップ。
- バックアップ一覧。
- 復旧手順。
- 復旧前確認。
- DB破損や誤操作時の初動手順。
- バックアップ運用ドキュメント。

### 完了状態

通常運用で予約データを保護でき、障害時に復旧手順を実行できる。

### 次に残すこと

- 履歴、監査、管理者向け可視化。

## Phase 8: 履歴、監査、変更追跡

### 目的

予約変更、削除、一時利用停止、管理者操作の経緯を追えるようにする。

### 主な実装範囲

- 予約履歴。
- PIN編集による変更履歴。
- 管理者一時利用停止の履歴。
- 影響を受けた予約の追跡。
- 履歴画面。
- 予約詳細からの履歴確認。

### 完了状態

予約がいつ、どのように変更されたかを確認できる。

### 次に残すこと

- 集計、利用状況の見える化。

## Phase 9: 利用状況、集計、管理者ビュー強化

### 目的

管理者がチャンバー利用状況を把握し、運用改善に使える情報を確認できるようにする。

### 主な実装範囲

- チャンバー別利用状況。
- 期間別予約数。
- ブロック使用傾向。
- キャンセル、削除、影響あり予約の把握。
- CSV出力。
- 管理者ダッシュボード。

### 完了状態

管理者が予約運用の混雑、空き、偏りを確認できる。

### 次に残すこと

- UX改善、検索精度改善、現場フィードバック反映。

## Phase 10: UX仕上げと現場運用調整

### 目的

実際の利用者、管理者の操作に合わせて、画面の迷い、入力の手間、候補表示のわかりにくさを解消する。

### 主な実装範囲

- 空き検索画面の整理。
- ブロック選択の操作性改善。
- 候補一覧の比較しやすさ改善。
- 予約ボードの視認性改善。
- スマートフォンや小型画面での確認性。
- エラー文言、確認文言の調整。
- ブラウザでの実利用シナリオ確認。

### 完了状態

利用者が説明を読まなくても、条件入力から予約確定まで迷わず進める。

### 次に残すこと

- 将来拡張の要否判断。

## Phase 11: 完全版リリース確認

### 目的

完全ゴールに対して、実装、運用、検証、文書が揃っていることを確認する。

### 主な確認範囲

- 温度サイクル型、固定条件型、自由温湿度型の予約。
- 複数チャンバー横断検索。
- 4 x 3 ブロック予約。
- 断片化なし候補生成。
- PIN編集、削除。
- 利用後削除不可。
- 管理者一時利用停止。
- 影響予約表示。
- 社内LAN運用。
- バックアップ、復旧。
- 履歴、集計。
- ブラウザ確認。

### 完了状態

ChamberReserve を社内の恒温槽予約管理サービスとして継続運用できる状態と判断できる。

## 検証ゲート

各フェーズの最後には、最低限次を確認する。

- そのフェーズの主要機能がブラウザで確認できる。
- 予約可否判定のテストがある。
- 競合や除外条件のテストがある。
- ビルドが通る。
- 既存フェーズの主要フローが壊れていない。
- 検証結果を `PASS` / `FAIL` / `BLOCKED` / `NOT_RUN` で残す。

## フェーズ開始文書の作成ルール

各フェーズを開始する前に、次の2つを作成する。

- `docs/CHAMBER_RESERVE_PHASEX_..._START.md`
- `docs/CHAMBER_RESERVE_PHASEX_..._GOAL.md`

開始文書には、少なくとも次を記載する。

- 目的。
- 背景。
- 読むべきファイル。
- 実装対象。
- 対象外。
- 完了条件。
- 検証コマンド。
- ブラウザ確認シナリオ。
- 次フェーズへの引き継ぎ。

`/goal` 文書は、開始文書を参照する短い実行プロンプトにする。
