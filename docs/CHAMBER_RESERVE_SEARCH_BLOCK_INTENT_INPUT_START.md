# ChamberReserve Search Block Intent Input Start

作成日: 2026-06-22

## 目的

空き検索の条件入力画面にある 4 x 3 ブロック選択を、「特定チャンバーの現在空き状況を見ながら選ぶ画面」から「複数チャンバー横断検索に渡す希望ブロック条件を入力する画面」へ修正する。

現在の条件入力画面では、初期状態で2ブロックが選択済みであり、さらに予約済みや一時停止の凡例と無効表示が出る。これは単一チャンバー前提では分かりやすい面があるが、Phase 3 以降の複数チャンバー候補検索では意味が合わない。

条件入力時点では、まだどのチャンバーのどの枠を予約するかは決まっていない。したがって、この画面で表示すべきなのは「希望するブロック数、形状、位置意図」であり、特定チャンバーの予約済み/一時停止状態ではない。

## 背景

ChamberReserve の検索入口はチャンバー選択ではなく条件検索である。ユーザーは、試験種別、希望日、サイクル数または利用時間、温湿度条件、所属部署、用途、そして必要なブロック条件を入力し、システムが条件に合うチャンバー候補を返す。

複数チャンバー検索では、同じ 4 x 3 ブロック指定であっても、候補生成時には TC-01、TC-02、TC-03 など各チャンバーの予約、一時停止、設定を別々に判定する。条件入力画面でひとつのチャンバーだけの予約済み/一時停止を見せると、次の誤解が起きる。

- あるチャンバーでは埋まっている場所でも、別チャンバーでは予約可能かもしれない。
- 一時停止表示が、全チャンバー共通の不可状態に見えてしまう。
- サイズだけ指定のとき、本来は場所を固定しないのに、空き状況を見ながら場所選びをしているように見える。
- 初期2ブロック選択が、システム推奨または必須条件のように見える。

このため、条件入力のブロックUIは「空き状況表示」ではなく「希望条件入力」へ責務を絞る。

## Core Product Decision

条件入力画面のブロック選択は、予約可否を表示する場所ではない。

表示するもの:

- 4 x 3 のチャンバー正面投影。
- ユーザーが選択した希望ブロック。
- 選択数。
- 連続選択であるか。
- `サイズのみ` / `位置まで指定` の意図。

表示しないもの:

- 予約済みブロック。
- 一時停止ブロック。
- 特定チャンバーの空き/不可状態。
- 候補生成前の使用予定ブロック。

予約済みや一時停止は、候補生成ロジックで判定し、候補表示画面、予約ボード、管理者一時利用停止画面で扱う。

## UX 方針

### 初期状態

初期表示では選択ブロックを0件にする。

理由:

- ユーザーが必要サイズを明示的に入力したことが分かる。
- 初期2ブロックが標準条件や推奨条件に見える誤解を防ぐ。
- 複数チャンバー検索で、入力条件と候補生成条件の関係が明確になる。

初期表示例:

```text
希望ブロック条件
未選択
```

検索ボタン押下時に0件なら、次のような明確なメッセージを出す。

```text
希望ブロックを1つ以上選択してください。
```

### ラベル

条件入力段階では、ブロックはまだ確定した利用場所ではない。そのため、画面上のラベルは次の方向へ寄せる。

推奨:

- `希望ブロック条件`
- `必要ブロック範囲`
- `ブロック条件`

避ける:

- `利用ブロック`
- `使用ブロック`
- `予約ブロック`

候補表示後の候補詳細や予約確定後は、実際に使う配置が決まるため `使用ブロック` や `候補ブロック` と表現してよい。

### 凡例

条件入力画面の凡例は最小限にする。

残す:

- `選択中`

削除する:

- `予約済み`
- `一時停止`

必要なら補助文で、空き判定は検索後に候補へ反映されることを伝える。

例:

```text
ここでは必要な形と大きさを指定します。予約済みや一時停止は候補生成時にチャンバーごとに判定します。
```

### サイズのみ

`サイズのみ` は、選択した形状とブロック数を満たす配置を各チャンバーから探すモードである。

条件入力画面では、選んだ場所そのものが空いているかは表示しない。候補生成時に、各チャンバー内で同じ形状が入る連続配置を探し、代表候補を表示する。

UI文言例:

```text
サイズのみ: 選択した形と大きさが入る候補を各チャンバーから探します。
```

### 位置まで指定

`位置まで指定` は、各チャンバーで同じ位置が空いている候補だけを返すモードである。

条件入力画面では、予約済み/一時停止を先に見せない。検索後に、その位置が空いているチャンバーだけが候補に出る。

UI文言例:

```text
位置まで指定: 各チャンバーで同じ位置が空いている候補だけを表示します。
```

## 状態管理方針

### 初期選択

`DEFAULT_SELECTED_BLOCKS` がある場合は、初期値を空配列にするか、検索フォーム初期化時に使わない。

推奨:

```ts
const DEFAULT_SELECTED_BLOCKS: BlockId[] = [];
```

または、より意図が明確な名前に変える。

```ts
const INITIAL_SEARCH_BLOCKS: BlockId[] = [];
```

### 新規検索

`新しく検索` や初期表示では、選択ブロックを空に戻す。

```ts
setSelectedBlocks([]);
```

### 条件変更

候補表示画面から `条件を変更` で戻る場合は、直前に検索したブロック条件を復元してよい。これは「新規初期状態」ではなく「検索済み条件の編集」だからである。

ただし、条件入力画面へ戻った時点でも予約済み/一時停止表示は出さない。

### 未選択時の検索

検索実行時に `selectedBlocks.length === 0` の場合は、候補生成を呼ばずに入力エラーとして扱う。

優先メッセージ:

1. ブロック未選択。
2. ブロックが連続していない。
3. その他必須項目不足。

例:

```ts
if (selectedBlocks.length === 0) {
  setSearchMessage('希望ブロックを1つ以上選択してください。');
  return;
}
if (!selectionIsContiguous) {
  setSearchMessage('ブロックは連続した範囲で選択してください。');
  return;
}
```

## 実装方針

### 1. 初期選択を空にする

対象候補:

- `src/App.tsx`
- `DEFAULT_SELECTED_BLOCKS`
- `useState<BlockId[]>`
- `startNewSearch`
- 新規検索や初期化処理

実施内容:

- 初期値を `[]` にする。
- `startNewSearch` でも `[]` をセットする。
- `条件を変更` で戻るときだけ、submittedSearch の選択ブロックを復元してよい。

### 2. 条件入力画面で unavailable blocks を表示しない

対象候補:

- `src/components/search/SearchFlow.tsx`
- 互換用に残っている場合は `src/App.tsx` 内の旧 `SearchConditionView`

実施内容:

- 条件入力画面の `BlockGrid` へ `disabledBlocks={unavailableBlocks}` を渡さない。
- 互換上 props が必要な場合も `disabledBlocks={[]}` にする。
- `unavailableBlocks` prop は条件入力画面に不要なら削除してよい。
- `App.tsx` 側で `unavailableBlocks` の計算が条件入力専用なら削除する。予約ボードや管理画面で使う場合は残す。

### 3. 凡例を条件入力用に整理する

対象候補:

- `SearchFlow.tsx`
- `Legend` 呼び出し

実施内容:

- `予約済み` 凡例を削除する。
- `一時停止` 凡例を削除する。
- `選択中` または `希望ブロック` の凡例だけにする。
- 必要なら補助文を追加する。

### 4. ラベルと説明文を修正する

対象候補:

- ブロック選択パネル title
- `BlockGrid` title
- header metric
- 空状態/未選択メッセージ

推奨変更:

```text
ブロック選択 -> 希望ブロック条件
利用ブロック -> 希望ブロック
選択 0ブロック -> 未選択
```

検索条件サマリーや候補詳細では、実候補に紐づくため `必要ブロック`、`候補ブロック`、`使用ブロック` を使ってよい。

### 5. 未選択時の検索メッセージを分ける

対象候補:

- `runSearch`
- `canSearch`
- `searchMessage`
- 入力バリデーション helper があればその関数

実施内容:

- `selectedBlocks.length === 0` を専用メッセージにする。
- 断片化と未選択を同じメッセージにしない。
- 検索ボタンを disabled にする場合でも、なぜ検索できないかが分かる表示を置く。

### 6. 候補表示側は維持する

候補表示、選択中候補、予約ボード、管理者一時利用停止では、候補ブロック、予約済み、一時停止、影響予約の表示が必要な場面がある。

今回削るのは条件入力画面の予約済み/一時停止表示だけであり、共通 `ChamberBlockMap` の状態表現や候補表示のブロック表示は削らない。

## テスト方針

### Domain

既存の候補生成ルールは原則変更しない。必要に応じて次を確認する。

- ブロック未選択では候補生成に進まない、または空候補になる。
- 断片化選択では候補が出ない。
- サイズだけ指定と位置まで指定の候補生成ルールは維持される。

### UI/Component

現状のテスト構成に合わせる。React component test がない場合は無理に追加しなくてよいが、可能なら次を確認する。

- 初期 `selectedBlocks` が空である。
- 条件入力画面の `BlockGrid` に予約済み/一時停止の disabled state が渡らない。
- 未選択時の検索メッセージが専用文言になる。

### Browser

必須確認:

- 初期表示でブロックが1つも選択されていない。
- 条件入力画面に `予約済み` と `一時停止` の凡例が出ない。
- 条件入力画面のブロックに予約済み/一時停止由来の無効色が出ない。
- 未選択で `候補を検索` を押すと `希望ブロックを1つ以上選択してください。` が出る。
- 1つ以上選択すると検索できる。
- `サイズのみ` では複数チャンバー候補が表示される。
- `位置まで指定` では指定位置が空いているチャンバーだけ候補に出る。
- 候補表示画面では候補ブロックが表示され、予約確定まで進める。
- 予約ボードと管理者一時利用停止のブロック表示は壊れていない。
- モバイル幅で条件入力のブロック選択と検索ボタンが操作不能に見切れない。

## チェックポイント

### Checkpoint 1: 現状の依存確認

- `selectedBlocks` 初期値と初期化処理を確認する。
- 条件入力画面で `unavailableBlocks` がどこから渡されているか確認する。
- 予約ボードや管理画面で同じ表示状態を使っていないか確認する。

完了条件:

- 条件入力だけから予約済み/一時停止表示を外す変更範囲が決まっている。

### Checkpoint 2: 初期未選択化

- 初期表示と新規検索の選択ブロックを空にする。
- 条件変更時の復元方針を実装する。

完了条件:

- 初期表示で選択済みブロックがない。
- 新しく検索でも選択済みブロックがない。

### Checkpoint 3: 条件入力ブロックUIの責務変更

- 条件入力画面の `BlockGrid` から予約済み/一時停止表示を外す。
- 凡例を選択中だけにする。
- ラベルを希望ブロック条件に寄せる。

完了条件:

- 条件入力画面で特定チャンバーの空き状態が表示されない。

### Checkpoint 4: 検索バリデーション

- 未選択時の専用メッセージを追加する。
- 断片化メッセージと必須項目不足メッセージを分ける。
- 1つ以上選択後は既存検索フローへ進める。

完了条件:

- 未選択で検索したときの理由が明確に表示される。
- 選択後は候補表示へ進める。

### Checkpoint 5: 検証と報告

- テスト、ビルド、launcher check を実行する。
- ブラウザで条件入力、候補表示、予約確定、予約ボード、管理者一時利用停止を確認する。

完了条件:

- 検証結果を PASS / FAIL / BLOCKED / NOT_RUN で報告できる。

## Completion Criteria

- 初期表示でブロックが選択されていない。
- `新しく検索` でもブロックが選択されていない。
- 条件入力画面では予約済み/一時停止の凡例や無効表示が出ない。
- 条件入力画面のブロックUIが、希望する形・大きさ・位置意図を指定するものとして読める。
- 未選択で検索すると、ブロック未選択の専用メッセージが出る。
- 1つ以上の連続ブロックを選ぶと候補検索できる。
- 複数チャンバー候補検索、候補ブロック表示、予約確定、PIN、予約ボード、管理者一時利用停止が壊れていない。
- テスト、ビルド、launcher check、ブラウザ確認を PASS / FAIL / BLOCKED / NOT_RUN で報告できる。
