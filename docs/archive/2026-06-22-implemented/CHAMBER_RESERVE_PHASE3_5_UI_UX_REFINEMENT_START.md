# ChamberReserve Phase 3.5 UI/UX Refinement Start

作成日: 2026-06-21

## 目的

Phase 3 で実装した複数温度サイクル型チャンバーの候補検索を、利用者が「このシステムはチャンバー予約のために作られている」と見た目から理解できる水準へ引き上げる。

本フェーズでは、生成した3案のうち第2案「候補比較 + 4 x 3 ブロック選択」を基準UIにする。汎用的なフォームとカードの集合ではなく、候補比較、25°C定常期間、運転期間、連続ブロック、予約確定を一体で判断できる画面に作り替える。

## 背景

Phase 1 から Phase 3 までで、空き検索、複数チャンバー候補、予約確定、4桁PIN編集/削除、予約ボード、管理者一時利用停止は動作確認できる状態になった。

一方で現在のUIは、業務ルールは満たしているが、視覚的には汎用テンプレートのフォーム/カードに見えやすい。Phase 4 で固定条件型チャンバーを追加すると候補比較の情報量がさらに増えるため、その前に検索結果画面とブロック選択体験を製品固有のUIとして固める。

## 長期計画上の位置づけ

- Phase 3 と Phase 4 の間に挿入するUI/UX強化フェーズ。
- 新しいチャンバー種別、DBスキーマ、API機能を増やすフェーズではない。
- Phase 4 以降の固定条件型/自由温湿度型を受け入れられるよう、候補比較の視覚構造を先に整える。

## 基準UI

第2案を基準にする。

- 左側または上部に検索条件の要約と再検索操作を置く。
- 中央に「予約可能候補のみ」を比較できる候補一覧を置く。
- 右側に大きな `4 x 3 ブロック選択` / 候補詳細パネルを固定する。
- 各候補には、チャンバー名、25°C搬入、運転期間、25°C搬出、使用ブロック、配置意図、予約アクションを表示する。
- 候補を選ぶと、右側のブロックパネル、タイムライン、確定ボタンが連動する。
- 予約確定後は、予約IDと4桁PINを同じ文脈で確認できるようにする。

## UI方針

### 1. 検索結果を主画面にする

空き検索は入口であり、候補比較がChamberReserveの中心価値である。検索フォームを画面の主役にしすぎず、検索後は候補比較とブロック判断へ視線が移る構成にする。

### 2. 候補はカードではなく比較行として扱う

候補は「装置 + 日時 + ブロック + 確定可否」の運用枠である。候補一覧は、汎用カードの羅列ではなく、比較しやすい行またはパネルとして設計する。

表示する情報:

- チャンバー名と場所。
- 25°C 搬入期間。
- 運転期間。
- 25°C 搬出期間。
- 使用ブロックのミニマップ。
- 配置種別: exact / size-only。
- 予約可能状態。
- 選択中状態。

### 3. 4 x 3ブロックを装置の正面投影として扱う

ブロック表示は単なるチェックボックス群ではなく、チャンバー内配置の正面投影として見せる。

必要な状態:

- 空き。
- 選択中。
- 候補で使用。
- 予約済み。
- 一時利用停止。
- 影響予約。
- 選択不可。

ブロックは常に 4列 x 3行の安定した寸法にし、ラベル、ホバー、選択、ドラッグ操作でレイアウトが揺れないようにする。

### 4. 25°C定常期間を候補比較の軸にする

温度サイクル型では、利用者が本当に確認したいのは「いつ搬入でき、いつ運転され、いつ搬出できるか」である。

各候補に以下を一貫して表示する:

- `25°C 搬入`
- `運転期間`
- `25°C 搬出`

この3区間は色と長さで区別し、単なる文章説明にしない。

### 5. 色の意味を固定する

色は装飾ではなく状態表現に使う。

- teal: 選択中、予約可能、主操作。
- amber: 注意、影響予約、確認が必要な状態。
- red: 一時利用停止、不可、削除/危険操作。
- gray: 予約済み、無効、履歴、補助情報。
- charcoal: 主文、見出し、操作ラベル。
- white / off-white: 作業面。

濃い青/紫グラデーション、ベージュ単色、装飾的な丸い背景、マーケティング風ヒーローは使わない。

## Scope

- 検索画面を第2案の候補比較UIへ作り替える。
- 候補一覧を比較しやすい行/パネルへ変更する。
- 選択中候補の詳細パネルを追加する。
- 大きな 4 x 3 ブロック選択/候補表示パネルを追加または再構成する。
- `CycleTimeline` の見え方を候補比較向けに調整する。
- `BlockGrid` を装置正面投影として見えるように調整する。
- 予約確定後の予約ID/PIN表示を候補選択の流れに自然に接続する。
- 予約ボード、PIN編集、管理者一時利用停止は既存機能を壊さず、色と余白の最低限の整合を取る。
- 必要なら `src/App.tsx` から検索結果周辺の小コンポーネントを切り出す。
- ブラウザ確認用スクリプトを第2案のUI構造に合わせて更新する。
- Phase 3.5 実装サマリーを作成する。

## Out of Scope

- 固定条件型チャンバー対応。
- 自由温湿度型チャンバー対応。
- サーバーAPIの新規機能追加。
- SQLiteスキーマ変更。
- 複数チャンバー登録CRUD。
- アカウント管理。
- 仮押さえ。
- 高度なPINセキュリティ。
- 通知連携。
- 利用率レポート。
- 本番運用ランタイムの拡張。

## 守るルール

- 最初の画面はチャンバー選択ではなく空き検索にする。
- 候補一覧には予約可能な候補だけを出す。
- 断片化したブロック割り当ては候補に出さない。
- 25°C搬入、運転期間、25°C搬出の見え方を弱めない。
- 4桁PINによる編集/削除フローを維持する。
- 利用後削除不可を維持する。
- 管理者一時利用停止の影響予約表示を維持する。
- TeamPlannerの業務概念を持ち込まない。
- URL、ポート、DB、ログ、PIDファイルの分離方針を妨げない。
- 既存の未コミット変更を戻さない。

## 実装チェックポイント

### Checkpoint 1: デザイントークンと画面骨格

- Tailwind設定またはCSSで、ChamberReserve用の色、線、余白、影、角丸を整理する。
- 画面全体を業務アプリとして落ち着いた密度にする。
- ナビゲーションはコンパクトに保ち、検索画面を主役にする。

完了条件:

- 初期表示が検索起点である。
- 画面が汎用ランディングページやヒーローに見えない。
- 主要な文字、ボタン、入力が重ならない。

### Checkpoint 2: 検索条件パネル

- 検索条件は左側または上部に整理する。
- 必須情報、ブロック数、配置意図、希望日を短時間で確認できるようにする。
- 検索後も条件を見失わず、再検索しやすくする。

完了条件:

- 検索条件の入力、変更、再検索が従来通り動く。
- invalid なブロック選択では検索できない。
- size-only / exact-location の意図が画面上で理解できる。

### Checkpoint 3: 候補比較リスト

- 候補一覧を比較行または比較パネルへ変更する。
- 各候補にタイムライン、チャンバー名、ブロックミニマップ、配置種別を表示する。
- 選択中候補を明確にする。
- 候補がない場合の空状態を簡潔に表示する。

完了条件:

- 複数チャンバー候補を比較できる。
- 候補一覧に予約不可能な候補が混ざらない。
- 候補を選択すると詳細パネルが連動する。

### Checkpoint 4: 4 x 3 ブロック詳細パネル

- 右側または候補詳細領域に大きな 4 x 3 ブロック表示を置く。
- 候補ブロック、選択ブロック、予約済み、利用停止、影響予約を状態として表示する。
- ブロックUIは固定寸法にし、ラベルや状態変化で崩れないようにする。

完了条件:

- 候補選択時に使用ブロックが大きなパネルに反映される。
- ユーザーが exact-location のブロック指定を変更できる。
- 断片化した選択は検索不可として見える。

### Checkpoint 5: 予約確定とPIN表示

- 選択中候補から予約確定へ進む主操作を明確にする。
- 予約確定後、予約IDと4桁PINを見逃さない位置に表示する。
- 予約編集への導線を維持する。

完了条件:

- 候補選択から予約確定まで迷わず進める。
- 予約確定後にPINが確認できる。
- PIN編集/削除フローへ遷移できる。

### Checkpoint 6: 既存機能の視覚整合

- 予約ボード、PIN編集、管理者一時利用停止の最低限の見た目を新しいトークンに合わせる。
- 管理者一時利用停止は影響予約が確認前に見える状態を維持する。

完了条件:

- 既存の主要フローが壊れていない。
- 一時利用停止の影響予約表示が維持される。
- Phase 4 へ進む前のUI基準として違和感がない。

### Checkpoint 7: ブラウザ確認と記録

- デスクトップとモバイル幅で確認する。
- スクリーンショットを `tmp_validation/` に残す。
- Phase 3.5 実装サマリーを作成する。

完了条件:

- `npm run test` が PASS。
- `npm run test:server` が PASS。
- `npm run build` が PASS。
- `python main.py --check` が PASS。
- `python main.py --no-open` が PASS。
- ブラウザで指定シナリオを確認し、PASS / FAIL / BLOCKED / NOT_RUN で報告できる。

## ブラウザ確認シナリオ

- 初期表示が空き検索である。
- 条件入力から候補検索できる。
- TC-01 / TC-02 / TC-03 の候補を比較できる。
- 候補一覧には予約可能候補だけが出る。
- 候補を選ぶと詳細パネルと 4 x 3 ブロックが連動する。
- exact-location のブロック選択を変更して再検索できる。
- 予約確定後に予約IDと4桁PINが表示される。
- PINで予約を開き、編集できる。
- PINで予約を削除できる。
- 利用後の予約は削除できない。
- 管理者一時利用停止で対象チャンバーを選び、影響予約を確認して登録できる。
- モバイル幅で横スクロールや文字重なりがない。

## 完了条件

- 第2案を基準にした候補比較 + 4 x 3 ブロック選択UIが実装されている。
- ChamberReserve固有の判断材料が検索結果画面の中心にある。
- 既存の予約、PIN編集/削除、利用後削除不可、管理者一時利用停止が壊れていない。
- テスト、ビルド、ランタイム確認、ブラウザ確認の結果を PASS / FAIL / BLOCKED / NOT_RUN で報告できる。

## Phase 4 Handoff

Phase 4 では固定条件型チャンバーを候補に追加する。Phase 3.5 完了時には、候補比較リストと候補詳細パネルが、チャンバー種別ごとの差分を表示できる余地を残していることを確認する。

想定される引き継ぎ項目:

- 候補行にチャンバー種別を追加する表示位置。
- 固定条件型の温湿度条件を候補詳細に出す位置。
- 温度サイクル型の 25°C 搬入/運転/搬出と、固定条件型の利用可能期間を比較するための表示整理。
- サーバー側候補判定結果をUIでどう説明するか。
