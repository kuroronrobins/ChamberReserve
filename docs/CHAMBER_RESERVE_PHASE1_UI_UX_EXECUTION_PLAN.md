# ChamberReserve Phase 1 UI/UX Execution Plan

作成日: 2026-06-21

## 目的

この文書は、Phase 1「単一温度サイクル型チャンバーのUI/UX基盤」を `/goal` で実装するための実行計画を定義する。

Phase 1 は要素が多いが、現時点では闇雲に複数の `/goal` に分割しない。1つの Phase 1 goal の中で、チェックポイントを区切って実装、検証、報告する。

ただし、途中で画面設計や操作体験の見直しが大きくなった場合は、その時点で現在のチェックポイントを完了または中断し、Phase 1-A / Phase 1-B のように分ける。

## Phase 1 の実装姿勢

Phase 1 は、サーバー実装前にユーザー側の体験を固めるフェーズである。

このフェーズでは、次を優先する。

1. 利用者が迷わない予約導線。
2. 4 x 3 ブロック選択のわかりやすさ。
3. 温度サイクル型の搬入、運転、搬出の見え方。
4. 候補一覧の比較しやすさ。
5. 予約確定、PIN確認、編集、削除の自然さ。
6. 管理者一時利用停止と影響予約表示の把握しやすさ。

このフェーズでは、次を作り込まない。

- サーバーAPI。
- SQLite永続化。
- live DB。
- 本番起動、停止、再起動スクリプト。
- サーバー側の最終競合判定。

## 実装チェックポイント

### Checkpoint 1: アプリ基盤と画面骨格

目的:

- React + Vite + TypeScript の最小アプリを作る。
- 最初の画面を空き検索にする。
- 主要画面のナビゲーションを用意する。

対象:

- 空き検索。
- 予約ボード。
- 予約編集。
- 管理者一時利用停止。

完了条件:

- ブラウザでアプリが開く。
- 最初に空き検索画面が表示される。
- 主要画面へ移動できる。
- ランディングページや説明中心の画面になっていない。

### Checkpoint 2: モックデータと予約判定の基礎

目的:

- UI確認に必要な単一温度サイクル型チャンバーのデータを定義する。
- 4 x 3 ブロック、予約、候補、PIN、一時利用停止をブラウザ内で扱えるようにする。
- 断片化なし候補生成の基礎を作る。

対象:

- 単一温度サイクル型チャンバー。
- 標準サイクル条件。
- 搬入可能 25°C 定常期間。
- サイクル運転期間。
- 搬出可能 25°C 定常期間。
- 既存予約サンプル。
- 一時利用停止サンプル。

完了条件:

- 4 x 3 ブロックの状態を表現できる。
- 予約済み、空き、選択中、一時利用停止を区別できる。
- 分割配置を候補にしない判定を確認できる。
- 25°C 定常期間外の開始、終了が候補に出ない。

### Checkpoint 3: 空き検索とブロック選択

目的:

- 利用者が試験条件と必要ブロックを入力できるようにする。
- チャンバーの正面図を直接操作して必要エリアを選べるようにする。

対象:

- 試験種別。
- 標準サイクル条件の表示。
- 希望日。
- 4 x 3 ブロック選択。
- サイズだけ指定。
- 場所まで指定。

完了条件:

- チャンバー選択をせずに検索を開始できる。
- ブロックをクリックまたはドラッグで選べる。
- 選択ブロック数がわかる。
- サイズだけ指定と場所まで指定の違いが画面上で自然に理解できる。
- 予約済み、利用停止中のブロックは選択できない、または明確に選択不可として見える。

### Checkpoint 4: 候補一覧と日程調整

目的:

- 条件に合う予約可能候補だけを表示する。
- 利用者が日程をずらしながら候補を比較できるようにする。

対象:

- 候補カード。
- 搬入可能時間。
- サイクル運転時間。
- 搬出可能時間。
- 使用ブロック表示。
- 前日、翌日、翌週などの日程移動。

完了条件:

- 予約可能な候補だけが出る。
- 断片化が必要な候補は出ない。
- 搬入、運転、搬出が視覚的に分かれている。
- 候補がない場合は、利用者が次の探索に進める表示になっている。
- 候補理由よりも、予約可能性と時間帯が先に読める。

### Checkpoint 5: 予約確定、PIN、編集、削除

目的:

- 予約を確定した体験を作る。
- 4桁PINを表示する。
- 共通の予約編集画面から変更、削除できる体験を作る。

対象:

- 予約確定確認。
- 予約完了表示。
- 4桁PIN発行。
- 予約識別情報。
- PIN入力。
- 利用開始前、利用中の変更、削除。
- 利用後削除不可。

完了条件:

- 予約確定時に予約内容とPINが見える。
- PINを使って編集画面に入れる。
- 予約内容を変更できる体験が確認できる。
- 利用開始前、利用中の削除が確認できる。
- 利用後の予約は削除できないことが明確に表示される。

### Checkpoint 6: 予約ボードと管理者一時利用停止

目的:

- 現場で現在と今後の利用状況を確認できるようにする。
- 管理者が一時利用停止を登録し、影響予約を把握できるようにする。

対象:

- 日別または週別の予約ボード。
- ブロック占有表示。
- 一時利用停止登録。
- 影響を受ける予約一覧。
- 影響あり状態。

完了条件:

- 予約済み、利用中、利用後、一時利用停止、影響ありを見分けられる。
- 管理者が一時利用停止の対象期間とブロックを指定できる。
- 既存予約と重なる場合、影響予約が事前に表示される。
- 一時利用停止で予約が自動削除されない体験が確認できる。

### Checkpoint 7: 仕上げ、ブラウザ確認、Phase 2 引き継ぎ

目的:

- UI/UXとしてPhase 1を完了できる状態にする。
- Phase 2 のサーバーAPI、SQLite永続化へ渡す境界を明確にする。

対象:

- 表示文言。
- レスポンシブ確認。
- ブラウザ確認シナリオ。
- テスト。
- ビルド。
- Phase 2 引き継ぎメモ。

完了条件:

- `npm run test` が実行され、結果が報告されている。
- `npm run build` が実行され、結果が報告されている。
- ブラウザで主要シナリオが確認されている。
- Phase 2 で実装すべきAPI、保存データ、サーバー側判定が整理されている。

## 細分化判断

Phase 1 は、原則として1つの `/goal` で実行する。

ただし、次のいずれかに当てはまる場合は、Phase 1 内部で分割する。

- Checkpoint 3 まででUI方針の大幅な見直しが必要になった。
- ブロック選択や候補表示の操作感に大きな再設計が必要になった。
- 実装量が増え、ブラウザ確認なしで進めるリスクが高くなった。
- Phase 1 の中でサーバー側に手を出す必要が出てきた。
- ユーザーが一度UIを触ってから方向修正したいと判断した。

分割する場合の候補は次の通り。

- Phase 1-A: アプリ基盤、モックデータ、空き検索、ブロック選択。
- Phase 1-B: 候補一覧、予約確定、PIN編集、予約ボード。
- Phase 1-C: 管理者一時利用停止、仕上げ、Phase 2引き継ぎ。

現時点では、この分割は予約枠として扱い、最初から別 `/goal` にはしない。

## ブラウザ確認シナリオ

Phase 1 の最後には、少なくとも次のシナリオをブラウザで確認する。

1. 空き検索画面を開く。
2. 標準サイクル条件を確認する。
3. 4 x 3 ブロックから連続した複数ブロックを選ぶ。
4. サイズだけ指定で候補を表示する。
5. 場所まで指定に切り替えて候補の変化を確認する。
6. 日程を前後にずらす。
7. 候補を選んで予約確定する。
8. 予約完了画面で4桁PINを確認する。
9. 共通の予約編集画面からPINで予約を開く。
10. 予約内容を変更する。
11. 利用開始前または利用中の予約を削除する。
12. 利用後予約が削除できないことを確認する。
13. 予約ボードで予約状態を確認する。
14. 管理者一時利用停止を登録する。
15. 影響を受ける予約が表示されることを確認する。

## Phase 2 へ渡す内容

Phase 1 完了時に、次をPhase 2へ渡す。

- 画面上で使用したモックデータの構造。
- 予約候補生成で必要になった入力項目。
- 予約確定に必要なデータ項目。
- PIN編集に必要な予約識別情報。
- 予約状態の種類。
- 一時利用停止の入力項目。
- 影響予約表示に必要なデータ。
- サーバー側で再判定すべきルール。

Phase 2 では、これらをもとにAPI、SQLite、最終競合判定、永続化後の再読み込みを実装する。
