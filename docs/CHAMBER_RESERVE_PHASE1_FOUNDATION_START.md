# ChamberReserve Phase 1 Foundation Start

## 目的

Phase 1 では、ChamberReserve の最小実用形を作る。

対象は温度サイクル型チャンバー1台とする。ただし、ユーザー体験は将来の複数チャンバー検索を前提にし、最初から「条件を入力して候補を探す」流れにする。

## 参照すべき決定事項

- 完成方針: `docs/CHAMBER_RESERVE_SYSTEM_SPEC.md`
- 実装運用方針: `AGENTS.md`
- `/goal` 方針: `docs/GOAL_IMPLEMENTATION_POLICY.md`

## Phase 1 の完成状態

利用者が次の一連の操作をブラウザ上で完了できる状態にする。

1. 空き検索画面を開く。
2. 温度サイクル型の標準試験条件を確認する。
3. 4 x 3 のチャンバー正面ブロックを視覚的に選ぶ。
4. サイズだけ指定、または場所まで指定を選ぶ。
5. 希望日を指定する。
6. 条件に合う予約候補を見る。
7. 候補を選んで予約を確定する。
8. 4桁PINを確認する。
9. 共通の予約編集画面から、予約識別情報とPINで予約を変更または削除する。
10. 管理者が一時利用停止を登録し、影響を受ける予約を確認できる。

## 実装対象

### UI

- 空き検索画面。
- チャンバー正面投影の 4 x 3 ブロック選択。
- サイズだけ指定と場所まで指定の切り替え。
- 候補一覧。
- 予約確定画面。
- 予約編集、削除画面。
- 予約ボード。
- 管理者一時利用停止画面。

### 予約ルール

- 温度サイクル型はユーザーが温湿度を変更できない。
- 予約開始と終了は 25°C 定常期間に限る。
- 標準試験条件を初期値とする。
- 通常予約の時間単位は1時間。
- 断片化したブロック候補は表示しない。
- 同じ枠にほぼ同時に予約が来た場合は、先に確定した予約を優先する。
- 仮押さえは作らない。
- 予約確定時には直前の競合を再確認する。
- 予約確定時に4桁PINを自動発行する。
- 利用開始前と利用中は変更、削除できる。
- 利用後は削除できない。

### 管理者操作

- 管理者は任意の期間、チャンバー全体または一部ブロックに一時利用停止を設定できる。
- 一時利用停止は通常予約より優先する。
- 既存予約と重なる場合、影響を受ける予約者、連絡先、試験名、時間、ブロックを表示する。
- 一時利用停止で影響を受けた予約は自動削除しない。

### データとして扱う概念

- チャンバー。
- チャンバー種別。
- 4 x 3 ブロック。
- 温度サイクル標準条件。
- 搬入可能な 25°C 定常期間。
- サイクル運転期間。
- 搬出可能な 25°C 定常期間。
- 予約。
- 予約状態。
- 4桁PIN。
- 一時利用停止。
- 影響を受ける予約。

## 対象外

Phase 1 では、次を実装しない。

- 複数チャンバーの実データ運用。
- 固定条件型チャンバー。
- 自由温湿度型チャンバー。
- 温湿度の許容幅。
- アカウント管理。
- 高度なPINセキュリティ。
- 仮押さえ。
- メール、Teams、Slack通知。
- 奥行き、高さ、重量、通風クリアランス管理。
- 自動バックアップや復元訓練。
- 高度な稼働率レポート。

## UI 方針

- 最初の画面は空き検索にする。
- チャンバー選択を最初の操作にしない。
- 画面説明文に頼らず、ブロック、色、状態、候補カードで理解できるようにする。
- 温度サイクル型では、搬入、運転、搬出を明確に分けて見せる。
- 候補に出るものは予約可能なものだけにする。
- 予約できない候補を表示して利用者に理由を考えさせない。

## 技術方針

TeamPlanner のローカル運用スタイルを踏襲する。

- React + Vite + TypeScript。
- Tailwind CSS。
- lucide-react。
- Node.js TypeScript API。
- SQLite。
- TeamPlanner と異なるURL、ポート、DB、ログ、PIDファイルを使う。

TeamPlanner のGantt、タスク、メンバー、休暇、認証、ロール、通知、バックアップなどの業務概念は持ち込まない。

## 検証方針

Phase 1 完了時には、少なくとも次を確認する。

- 予約候補生成のドメインテスト。
- 断片化候補を出さないことのドメインテスト。
- 25°C 定常期間外の開始、終了を拒否するテスト。
- 予約確定時の競合再確認テスト。
- PINによる編集、削除テスト。
- 利用後予約の削除拒否テスト。
- 管理者一時利用停止と影響予約表示のテスト。
- `npm run test`。
- `npm run build`。
- ブラウザで、空き検索、ブロック選択、予約確定、PIN編集、管理者一時利用停止まで確認する。

## 完了条件

Phase 1 は、次の状態で完了とする。

- 単一温度サイクル型チャンバーの予約フローがブラウザ上で通る。
- 4 x 3 ブロック選択と断片化なし候補生成が機能する。
- 25°C 定常期間ルールがUIとサーバー側の両方で守られる。
- 4桁PINで予約変更、削除ができる。
- 利用後予約は削除できない。
- 管理者一時利用停止が通常予約より優先され、影響予約を表示できる。
- テスト、ビルド、ブラウザ確認の結果が `PASS` / `FAIL` / `BLOCKED` / `NOT_RUN` で報告されている。

## 次フェーズ候補

Phase 1 後の候補は次の通り。

- Phase 2: 複数チャンバー候補検索。
- Phase 3: 固定条件型チャンバー対応。
- Phase 4: 自由温湿度型チャンバー対応。
- Phase 5: 社内LAN運用、バックアップ、復旧手順。
- Phase 6: 利用率、履歴、管理者向け集計。
