# ChamberReserve Goal Implementation Policy

作成日: 2026-06-20

この文書は、ChamberReserve の実装を Codex の `/goal` で進めるための標準方針を定義する。

## 1. 基本方針

ChamberReserve の実装は、通常の1ターン作業ではなく、Markdownの開始契約に基づく `/goal` を標準とする。

対象:

- コード変更。
- 設定変更。
- サーバー起動、停止、検証スクリプトの追加。
- DB、予約、チャンバー、PIN、一時利用停止、同時操作に関わる実装。
- UI、UX、ブラウザ確認を伴う実装。
- 運用手順や設計Markdownの追加、修正。
- テストや検証コマンドの整備。

通常ターンでよいもの:

- 質問への回答。
- 方針相談。
- 調査のみ。
- コードレビュー。
- 状況確認。
- ユーザーが明示的に「通常実行」と指定した作業。

## 2. `/goal` の分割方針

1つの `/goal` には、1つの目的だけを持たせる。

よい単位:

- Phase 1 のアプリ基盤と単一温度サイクルチャンバー予約。
- 予約検索ロジックの強化。
- 管理者一時利用停止。
- 複数チャンバー候補検索。
- ランタイム運用とバックアップ。

避ける単位:

- UI、DB、複数チャンバー、バックアップ、通知、認証をまとめて一度に実装する。
- 初期MVPに不要な高度なセキュリティを混ぜる。
- 将来フェーズの自由温湿度型や固定条件型を、単一温度サイクル型MVPに無理に入れる。

## 3. `/goal` プロンプトの基本構造

詳細はフェーズ開始Markdownに置き、`/goal` は短く保つ。

標準構造:

```text
/goal ChamberReserve の [対象フェーズまたは対象機能] を [具体的な成果] まで進めてください。

# Objective
[今回の目的を1つだけ書く]

# Read First
- AGENTS.md
- docs/CHAMBER_RESERVE_SYSTEM_SPEC.md
- docs/GOAL_IMPLEMENTATION_POLICY.md
- [対象フェーズ開始Markdown]

# Scope
- [変更してよい範囲]
- [実装対象]

# Out of Scope
- [今回やらないこと]

# Constraints
- 既存の未コミット変更を戻さない。
- 仕様書の決定事項を破らない。
- 実装範囲を現在のフェーズに限定する。

# Validation
- [必須検証コマンド]
- [ブラウザ確認]

# Progress
- チェックポイントごとに、変更内容、検証結果、残件、ブロック有無を短く報告する。

# Stop When
- [完了とみなす検証可能な状態]
```

## 4. 必須の完了報告

完了時は、次を明確に報告する。

- 変更ファイル。
- 実装した範囲。
- 実装しなかった範囲。
- 検証結果を `PASS` / `FAIL` / `BLOCKED` / `NOT_RUN` で分類。
- ブラウザ確認の有無。
- 残件。
- 次フェーズに渡すべき内容。

## 5. ブロック時の報告

ブロック時は、次だけを簡潔に報告する。

- ブロック理由。
- すでに試したこと。
- 現在壊れているか、未実装なだけか。
- ユーザー判断が必要な選択肢。

## 6. ChamberReserve 固有の守るべき境界

- 最初の画面は条件検索であり、チャンバー選択ではない。
- 初期対象は温度サイクル型1台。
- 4 x 3 はチャンバー全体の正面投影。
- 断片化したブロック割り当ては候補に出さない。
- 仮押さえは作らない。
- 4桁PINは簡易方式のままにする。
- アカウント管理は作らない。
- 管理者一時利用停止は通常予約より優先する。
- メール、Teams、Slack連携は作らない。
- 奥行き、高さ、重量、通風クリアランス管理は初期対象外。

## 7. フェーズ開始Markdownに含める内容

フェーズ開始Markdownには、少なくとも次を含める。

- 目的。
- 背景。
- 最初に読むファイル。
- 実装対象。
- 対象外。
- データや画面で守るべき仕様。
- 検証コマンド。
- ブラウザ確認の観点。
- 完了条件。
- 次フェーズ候補。
