# ChamberReserve Search Flow Split Start

作成日: 2026-06-22

## 目的

ChamberReserve の空き検索 UI を、条件入力画面と候補表示画面に明確に分割する。

現在の検索 UI は、入力中の条件、ブロック選択、候補一覧、候補詳細、予約確定操作が同じ画面に並びやすい。そのため、画面に表示されている候補が「いま編集中の条件」に対するものなのか、「直前に検索実行した条件」に対するものなのかが分かりにくい。

この改善では、ユーザーが条件を編集している状態と、提出済み条件に対する候補を比較している状態を UI と状態管理の両方で分離する。候補一覧には、明示的に検索実行された条件に対応する予約可能候補だけを表示する。

## 背景

ChamberReserve の最初の画面はチャンバー選択ではなく空き検索である。ユーザーは試験種別、希望日、利用時間またはサイクル数、使用ブロック、所属部署、用途などを入力し、条件に合う候補を確認して予約する。

一方で、検索条件と候補を同時に表示すると次の問題が出る。

- 条件を少し変更しただけで、下部の候補が現在条件に追随しているように見える。
- 実際には古い条件に対する候補であっても、その違いが画面上で伝わりにくい。
- サイクル試験と温湿度指定の入力モード差が、候補表示や予約確定操作と混ざって見える。
- モバイルや狭い画面では、条件入力と候補一覧の両方を追う必要があり、現在の操作対象が曖昧になる。

このため、検索フローを「条件入力」から「候補表示」へ進む明確なステップ型 UI にする。

## 基本方針

### 画面ステップ

検索フローは次の 2 ステップを基本にする。

1. 条件入力
2. 候補表示

必要であれば予約確定前の確認パネルまたはモーダルを候補表示内に置いてよい。ただし、この改善の主目的は条件入力と候補表示の分離であり、予約確定を独立ページ化することは必須ではない。

### 状態モデル

検索 UI の状態は、少なくとも次の 3 種類を分ける。

```ts
type SearchStep = "conditions" | "results";

interface SearchFlowState {
  step: SearchStep;
  searchDraft: SearchDraftState;
  submittedSearch: SubmittedSearchState | null;
  candidateResults: ReservationCandidate[];
  selectedCandidateId: string | null;
}
```

`searchDraft` は、条件入力画面で編集中の値である。入力中に候補一覧を再生成して表示しない。

`submittedSearch` は、ユーザーが「候補を検索」した時点の検索条件スナップショットである。候補表示画面の見出し、条件サマリー、候補一覧は必ずこの値に基づく。

`candidateResults` は、`submittedSearch` に対して生成または取得された候補だけを持つ。`searchDraft` の変更で暗黙に変わらない。

### 候補表示の原則

- 条件入力画面では候補一覧を表示しない。
- 候補表示画面には、検索実行済み条件のサマリーを必ず表示する。
- 候補一覧は `submittedSearch` と一対一に対応する。
- 候補が 0 件でも、候補表示画面に遷移して「この条件では候補がありません」と表示する。
- 条件を変更する場合は条件入力画面に戻す。
- 条件入力画面に戻った時点では古い候補を表示しない。
- 再検索するまで、新しい条件に対する候補が存在するように見せない。

## 画面設計

### 条件入力画面

目的は、検索条件を迷わず完成させることである。候補比較や予約確定の要素は置かない。

表示する主な内容:

- 試験種別の選択
  - サイクル試験
  - 温湿度指定
- サイクル試験の場合
  - 希望日
  - ステップ 1 開始時刻
  - サイクル数
  - 自動計算された終了日時
  - チャンバー側に登録された標準サイクル条件の概要
- 温湿度指定の場合
  - 開始日時
  - 利用時間
  - 自動計算された終了日時
  - 温度、湿度入力
- 使用ブロック数またはブロック選択
- 所属部署プルダウン
- 用途またはメモ
- 入力エラー、未入力項目、検索不可理由
- 「候補を検索」ボタン

条件入力画面の下部に候補カードを置かない。補助情報として予約ルールやチャンバー仕様の短い説明を置く場合も、候補一覧と誤認されない表現にする。

### 候補表示画面

目的は、提出済み条件に対する候補を比較し、予約する候補を選ぶことである。条件編集フォーム全体は置かない。

表示する主な内容:

- 画面見出し: `この条件で検索した候補`
- 検索条件サマリー
  - 試験種別
  - 期間
  - サイクル数または利用時間
  - 必要ブロック
  - 所属部署
  - 温湿度指定の場合は指定温湿度
- 候補一覧
- 選択中候補の詳細
  - 開始日時
  - 終了日時
  - 利用ブロック
  - 4 x 3 ブロック表示
  - 競合や一時停止の影響がないこと
- 予約確定操作
- `条件を変更` ボタン
- `新しく検索` ボタン
- 必要な場合のみ、日付を前後にずらした再検索操作

候補がない場合も、検索条件サマリーは残す。空状態には、候補が出なかった理由と次の行動を表示する。

例:

- 条件を変更
- 希望日をずらして再検索
- 使用ブロック数を減らす
- 利用時間またはサイクル数を調整する

## 推奨コンポーネント構成

既存構成に合わせて `src/components` 配下で整理する。既に feature ディレクトリがある場合は、その構成に合わせてよい。

推奨パス:

```text
src/components/search/SearchFlow.tsx
src/components/search/SearchConditionView.tsx
src/components/search/SearchResultsView.tsx
src/components/search/SearchConditionSummary.tsx
src/components/search/CandidateList.tsx
src/components/search/CandidateDetailPanel.tsx
```

### `SearchFlow.tsx`

検索フロー全体の状態を持つ薄いコンテナにする。

責務:

- `searchDraft` の保持
- `submittedSearch` の作成
- 候補生成または既存検索関数の呼び出し
- `conditions` / `results` の画面切り替え
- 予約確定後の必要な状態更新

`App.tsx` に大きな条件分岐や検索状態を残しすぎない。

### `SearchConditionView.tsx`

条件入力専用の画面にする。

責務:

- 入力フォームの表示
- サイクル試験と温湿度指定の入力差分表示
- 4 x 3 ブロック選択
- 所属部署プルダウン
- 入力値の即時バリデーション表示
- 検索実行ボタン

このコンポーネントは候補一覧を受け取らない。

### `SearchResultsView.tsx`

候補表示専用の画面にする。

責務:

- `submittedSearch` のサマリー表示
- 候補一覧表示
- 選択候補の詳細表示
- 予約確定操作
- 条件変更、再検索、新規検索の操作

このコンポーネントはフル編集フォームを持たない。変更したい場合は条件入力画面へ戻す。

### `SearchConditionSummary.tsx`

候補表示画面の上部に置く、提出済み条件の読み取り専用サマリーにする。

責務:

- 候補がどの条件に対するものかを一目で示す。
- サイクル試験と温湿度指定で表示項目を出し分ける。
- 編集可能な入力欄に見せない。

### `CandidateList.tsx`

候補カードのリスト表示を担当する。

責務:

- 予約可能候補のみ表示
- 選択中候補の強調
- 候補 0 件時の空状態
- 候補選択イベント

### `CandidateDetailPanel.tsx`

選択中候補の詳細表示と予約確定操作を担当する。

責務:

- 4 x 3 ブロック利用範囲の表示
- 期間、チャンバー、条件整合性の表示
- 予約確定前の必要情報表示
- 予約確定アクション

## データ変換方針

### `SearchDraftState`

UI 入力に近い形で持つ。未入力や編集中の値を許容する。

例:

```ts
interface SearchDraftState {
  mode: "cycle" | "environment";
  desiredDate: string;
  cycleStartTime: string;
  cycleCount: number;
  environmentStartDate: string;
  environmentStartTime: string;
  environmentDurationHours: number;
  temperatureCelsius: number | null;
  humidityPercent: number | null;
  selectedBlockIds: string[];
  departmentId: string;
  purpose: string;
}
```

既存の型がある場合は重複定義せず、既存型を整理して使う。

### `SubmittedSearchState`

検索実行時点で、候補生成に使える正規化済みデータと、画面表示用のサマリーを持つ。

例:

```ts
interface SubmittedSearchState {
  id: string;
  request: SearchRequest;
  summary: SearchConditionSummaryModel;
  submittedAt: string;
}
```

`id` は React の再描画や結果紐づけに使うローカル ID でよい。サーバー永続化 ID ではない。

### 候補生成

候補生成は `submittedSearch.request` を入力にする。`searchDraft` を直接参照しない。

これにより、候補表示中に条件入力画面へ戻って値を変更しても、古い候補が新しい条件に対するものとして表示されることを防ぐ。

## 操作フロー

### 初期表示

1. `step = "conditions"`
2. 条件入力画面を表示する。
3. 候補一覧は表示しない。

### 検索実行

1. `searchDraft` を検証する。
2. 入力エラーがあれば条件入力画面に表示する。
3. 問題なければ `submittedSearch` を作成する。
4. `submittedSearch.request` から候補を生成する。
5. `candidateResults` を更新する。
6. `selectedCandidateId` を初期化または先頭候補に設定する。
7. `step = "results"` にする。

### 条件変更

1. 候補表示画面で `条件を変更` を押す。
2. `step = "conditions"` に戻す。
3. `searchDraft` には前回の検索条件を編集しやすい形で復元する。
4. 候補一覧は表示しない。

`submittedSearch` と `candidateResults` を内部に残してもよいが、条件入力画面では使わない。実装が単純になるなら、条件変更時に `submittedSearch = null`、`candidateResults = []` にしてもよい。

推奨は、古い候補の誤表示を避けるため、条件入力画面に戻る時点で候補表示用状態をクリアすること。

### 新しく検索

1. 候補表示画面で `新しく検索` を押す。
2. 初期値または現在の既定値で `searchDraft` を作る。
3. `submittedSearch = null`、`candidateResults = []` にする。
4. `step = "conditions"` にする。

### 日付をずらして再検索

既存 UI に日付シフトがある場合は、候補表示画面に残してよい。ただし、操作結果は新しい `submittedSearch` として扱う。

1. 候補表示画面で日付シフトを実行する。
2. 現在の `submittedSearch.request` を基に日付だけ変更した新しい検索条件を作る。
3. 新しい `submittedSearch` を作成する。
4. 候補を再生成する。
5. 条件サマリーも新しい日付に更新する。

日付シフト後も、候補一覧と条件サマリーが一致していることを必ず保つ。

## UI/UX 要件

### 見出し

条件入力画面の見出しは、入力を進める画面であることが分かる表現にする。

例:

- `空き検索`
- `試験条件を入力`
- `必要な期間とブロックを指定`

候補表示画面の見出しは、検索結果であることが分かる表現にする。

例:

- `この条件で検索した候補`
- `予約可能な候補`

### ステップ表示

ヘッダーまたは画面上部に軽いステップ表示を置く。

```text
1 条件入力  >  2 候補選択  >  3 予約確定
```

ただし、過度に大きなヒーロー表現にしない。業務用予約サイトとして、現在位置が分かる小さく明瞭な表示にする。

### 条件サマリー

候補表示画面の上部に、提出済み条件を compact に表示する。

サイクル試験:

- 試験: サイクル試験
- 開始: 希望日とステップ 1 開始時刻
- サイクル数
- 終了予定
- 必要ブロック
- 所属部署

温湿度指定:

- 試験: 温湿度指定
- 開始
- 利用時間
- 終了予定
- 温度
- 湿度
- 必要ブロック
- 所属部署

### アクション配置

候補表示画面では、主ボタンを候補予約に置く。

主なアクション:

- `この候補で予約する`
- `条件を変更`
- `新しく検索`

`条件を変更` は目立たせすぎないが、必ず見つけやすくする。

### 空状態

候補 0 件時は、候補一覧の空白だけにしない。

表示する内容:

- この条件では予約可能候補がないこと
- 検索した条件のサマリー
- 次に試せる操作

## 実装スコープ

含めるもの:

- 検索 UI の条件入力画面と候補表示画面の分割
- `searchDraft` と `submittedSearch` の状態分離
- 候補一覧を提出済み条件に紐づける修正
- 条件変更、新規検索、日付シフト再検索の遷移整理
- 候補 0 件時の表示
- 既存予約確定フローとの接続維持
- 必要な範囲のコンポーネント分割
- 必要な範囲のテスト修正または追加

含めないもの:

- サーバー API の新設
- SQLite 永続化の変更
- URL ルーティングの導入
- アカウント管理
- 仮押さえ
- 通知連携
- 新しいチャンバー種別の業務ルール追加
- TeamPlanner の業務概念、タスク、ガント、休暇、メンバー管理の移植

## チェックポイント

### 1. 現状検索状態の棚卸し

- `App.tsx` と検索関連コンポーネントの状態を確認する。
- 入力中の値、検索実行済みの値、候補一覧、選択候補、予約確定入力を分類する。
- 変更対象を最小限に決める。

完了条件:

- どの状態を `searchDraft`、`submittedSearch`、`candidateResults` に分けるか決まっている。

### 2. 条件入力画面の分離

- 条件入力 UI を `SearchConditionView` に寄せる。
- 候補一覧を条件入力画面から外す。
- 検索ボタンでのみ候補生成へ進む。

完了条件:

- 初期表示で候補一覧が表示されない。
- 条件編集中に古い候補が見えない。

### 3. 候補表示画面の分離

- `SearchResultsView` を作る。
- `SearchConditionSummary` を表示する。
- 候補一覧と選択候補詳細を表示する。
- 予約確定操作を既存フローと接続する。

完了条件:

- 候補表示画面で、候補がどの条件に対するものか明確に分かる。
- 候補一覧は提出済み条件に対応している。

### 4. 遷移と再検索の整理

- `条件を変更` で条件入力画面に戻る。
- `新しく検索` で初期状態から検索を始められる。
- 日付シフト再検索がある場合は、新しい提出済み条件として扱う。

完了条件:

- 古い候補が新しい条件に対する候補のように見えない。
- 再検索後のサマリーと候補が一致している。

### 5. 検証と報告

- テストとビルドを実行する。
- ブラウザで主要フローを確認する。
- PASS / FAIL / BLOCKED / NOT_RUN で結果を整理する。

## ブラウザ確認項目

- 初期表示が条件入力画面であり、候補一覧が出ていない。
- サイクル試験で希望日、ステップ 1 開始時刻、サイクル数を入力して検索できる。
- 温湿度指定で開始日時、利用時間、温度、湿度を入力して検索できる。
- 検索後、候補表示画面に提出済み条件サマリーが表示される。
- 候補一覧が予約可能候補だけを表示する。
- 候補 0 件時に空状態と条件変更導線が表示される。
- `条件を変更` で条件入力画面に戻り、候補一覧が消える。
- 条件変更後に再検索すると、サマリーと候補が新しい条件になる。
- 候補選択から予約確定まで進める。
- PIN 編集/削除、利用後削除不可、管理者一時利用停止への既存導線が壊れていない。
- モバイル幅で左右ペインや下部操作が見切れず、現在画面が分かる。

## 完了条件

- 条件入力画面と候補表示画面が視覚的にも状態管理上も分離されている。
- 候補一覧は明示的に検索実行された条件に対してだけ表示される。
- 候補表示画面には提出済み条件サマリーが必ず表示される。
- 条件入力画面では候補一覧が表示されない。
- 条件変更後に古い候補が誤って表示されない。
- 予約確定、PIN 編集/削除、管理者一時利用停止の既存 UX が破綻していない。
- 検証結果を PASS / FAIL / BLOCKED / NOT_RUN で報告できる。
